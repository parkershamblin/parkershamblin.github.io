## Full Medium-Ready Draft

I am Parker Shamblin, a Computer Science student at the University of South Florida, and I am building a software engineering portfolio around a simple idea: projects should be understandable, useful, and honest about what they demonstrate.

A repository is valuable, but a repository alone does not always explain the engineering decisions behind the code. I want each project in my portfolio to make those decisions easier to see. That means documenting the problem, the architecture, the data flow, the tradeoffs, and the next improvements I would make.

## My Current Technical Focus

My portfolio work currently centers on three areas:

- Java backend development with structured application layers
- SQL databases that model practical workflows
- Python and OpenCV experiments that turn image input into visible results

These areas give me different kinds of engineering practice. Java and SQL projects push me to think about application structure, database relationships, validation, and maintainability. OpenCV projects push me to think about inputs, preprocessing, detection logic, debugging, and responsible documentation.

## A Java Backend Project With Real Workflows

One repository I am documenting presents a Retail Banking and Brokerage Platform learning project using Java, JSP, Jakarta Servlets, JDBC, Oracle XE, and Apache Tomcat.

Its documentation outlines public browsing and customer, teller, and manager workflow areas. It also maps those areas to database records such as account activity, transfers, trade orders, approvals, watchlists, and portfolio records.

What matters most to me is not presenting the project as something larger than it is. The useful part is the chance to explain how a multi-page Java web application can separate presentation, request handling, data access, and database logic in a way that stays readable.

You can review the repository here:

https://github.com/parkershamblin/retail-banking-brokerage-platform

## An OpenCV Project With Visible Feedback

Another repository in my portfolio describes a Python and OpenCV experiment built around a cascade-classifier workflow and desktop-window capture for object detection in a game environment.

Computer vision is useful for learning because the feedback is visible. When a detection is wrong, I can inspect the input image, the classifier output, the surrounding conditions, and the assumptions in the pipeline. That turns debugging into a concrete process instead of a vague one.

I keep the project framed as an experiment with clear limitations. The value is in understanding the image-processing workflow, documenting the constraints, and identifying the next measurable improvements.

You can review the repository here:

https://github.com/parkershamblin/opencv-OSRS1

## Documentation Is Part Of The Engineering Work

I want each project to answer a few practical questions quickly:

1. What problem does this project explore?
2. Which technologies does it use?
3. How does data move through the system?
4. What design decisions are visible in the code?
5. What limitations still need work?
6. How could the project be measured more honestly?

For backend projects, that can mean architecture notes, schema diagrams, screenshots, and a straightforward setup guide. For computer vision work, that can mean sample inputs, result images, pipeline diagrams, and a list of known constraints.

Clear documentation is not decoration. It helps another engineer evaluate the work, and it helps me see what I actually understand.
